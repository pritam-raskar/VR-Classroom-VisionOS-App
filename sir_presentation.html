Overview
What we're building

What is SIERA?

⚠️

Financial institutions process thousands of payment messages daily. Each message is screened for sanctions hits — potential matches against restricted entities. Today, analysts manually review every hit, which is time-consuming and error-prone.

🤖

The Solution

SIERA is an AI-powered orchestration layer that sits above the screening engine and uses LLM agents to analyse each sanctions hit — automatically determining whether it is a genuine threat or a false positive.

📨

Message Ingestion

Receives payment events from MQ. Supports both SWIFT MT messages and Fircosoft Plaintext formats, automatically detecting which format each message uses.

🧠

AI Delivery

A specialised LLM agent pipeline performs individual analysis of each sanctions hit — checking entity name, type, nationality, and context to make an informed recommendation.

📊

Analytics Engine

Real-time metrics, evaluation of LLM decisions vs. human analyst outcomes, latency monitoring, and risk management dashboards for compliance teams.

Primary Agent

First-Pass Analysis

Analyses each sanctions hit by entity type (Company, Individual, Vessel, Critic). Checks name, DOB, nationality, and payment context. Returns RELATED or CLEARED with reasoning.

Critic Agent

Secondary Validation

Receives the Primary Agent's decision and refines it. Verifies whether the beneficiary in the payment is genuinely the sanctioned entity or a false positive match.

FME Rule Engine

Auto-Disposition

Known false positive patterns (FME rules) are matched before LLM processing. Matching hits are auto-released instantly, saving LLM compute for truly ambiguous cases.

Process
End-to-end flow

How It Works

1

Parse & Fetch

Incoming alert is parsed. Payment message is fetched and validated. Supports SWIFT MT and Fircosoft Plaintext formats.

2

Construct LLM Request

Each hit is enriched with entity data, payment context, and risk tags (OTN, TXI = High Risk; BNK = Low Risk) before being sent to agents.

3

Parallel Processing

Valid hits are partitioned into batches (default: 3 hits per batch) and processed in parallel by specialised LLM agents.

4

Compute & Persist

Dispositions are calculated at hit and transaction level. Results are persisted and a recommended disposition is returned.
